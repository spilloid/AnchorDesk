Why materialticket

A helpdesk and an operations console, in one place.

Run it standalone or plug in your stack. Every feature works whether or not an external system is connected.

Local-first ticketing

Full CRUD, statuses, priorities, assignees, time entries, and a Kanban board — no external dependency required.

Full audit trail

Every change writes a before/after snapshot to an append-only log. Per-ticket revision history and attribution, built in.

Device inventory

LAN probes push discovered devices into your database. Link them to tickets so a card shows the machine — and whether it's up.

Act on machines

Queue and schedule scripts against a device through your RMM — Tactical RMM today — directly from the ticket.

OIDC authentication

Auth is delegated to any OIDC provider — Azure AD, Authentik, and more. No passwords ever stored locally.

Pluggable sync & MCP

Strategy/Repository boundaries make ConnectWise, IMAP-to-ticket, and RMM runners drop-in. A built-in MCP server lets AI agents manage tickets.

How it works

Your database is the core. Everything else is an adapter.

  1. 1

    Tickets land locally

    Created in the UI, emailed in, or synced from ConnectWise — every ticket is written to your MariaDB first.

  2. 2

    Probes report devices

    A scanner on the customer LAN heartbeats and pushes device records, upserted into your devices table.

  3. 3

    Link & act

    Attach devices to a ticket and run RMM scripts against them — run-now or scheduled — without leaving the ticket.

  4. 4

    Everything is audited

    Each mutation appends a before/after snapshot, giving you complete, attributable history.

# 1 · start the database + DB browser
docker compose up -d db adminer

# 2 · configure & push the schema
cp backend/.env.example backend/.env
cd backend && npx prisma db push

# 3 · run backend + frontend
npm start          # :8060
cd ../web-client && npm run dev   # :5173
